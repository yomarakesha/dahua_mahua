# DSS: стабильность и производительность — диагноз и план действий

**Дата:** 2026-06-16
**Статус:** рабочий план (3 уровня)
**Основано на:** `netcheck-result.md`, `docs/audit-plan.md`, ревью кода стриминга, документации MediaMTX.

---

## 0. Главный вывод (TL;DR)

Нестабильность (заикания, фризы, переподключения) **в основном вызвана тем, что
RTSP-ретранслятор main-потока на NVR Dahua теряет кадры** — это **не сеть, не
MediaMTX и не код DSS**. Это доказывают ваши же замеры `netcheck`.

| Наблюдение (netcheck) | Что исключает |
|---|---|
| Напрямую с камеры (main): **0** потерь. Через NVR та же камера, тот же кабель: **7815** | сеть, кабель, MediaMTX |
| Даже **один** main через NVR теряет ~5586 пакетов | суммарную нагрузку/насыщение — проблема в каждом потоке |
| 8 **sub** через тот же NVR: **0** потерь | NVR ретранслирует sub чисто; сломан только путь **main** |
| Потери при 6.5 Мбит/с на канале 100 Мбит/с | пропускную способность — гигабит не поможет |
| Тест шёл по **TCP** | сетевую потерю: TCP пересылает заново, значит **сам NVR не отдал кадры** |

**Почему это видно как фризы:** main Dahua обычно идёт SmartCodec (H.264+/H.265+)
с длинным GOP. NVR теряет P-кадр → картинка замирает до следующего I-кадра
(1–4 с). Переподключения — это срабатывание детектора зависаний (8 с) или закрытие
on-demand источника в MediaMTX.

**Честная оговорка:** настройка MediaMTX/WebRTC **не лечит** потери на стороне NVR.
Стабильность реально появляется на Уровне 1. Уровни 2–3 убирают вторичные эффекты
(много зрителей, шторм переподключений), но не заменяют Уровень 1.

**Важная поправка (живой аудит 2026-06-16):** прямой захват с камеры применим
**не ко всем** камерам на этом объекте — часть камер напрямую доступна (→ прямой
захват, 0 потерь), часть нет (→ остаётся на релее). Поэтому решение — прямой
захват **с проверкой доступности** (уже реализовано вторым разработчиком,
см. §1.1), а не «прямой захват всем подряд». Замеры `netcheck` с `192.168.23.11`
(0 потерь напрямую) были сделаны на **другом/старом** объекте.

## Уровень 1 — Архитектура (устраняет ~90%)

### 1.1. Прямой захват main — НО только там, где камера реально доступна

Код уже умеет это (`backend/app/services/path_sync.py:67` →
`_build_path_config`): если у камеры заполнен `Camera.ip`, main-поток тянется
**прямо с камеры** (канал 1, порт 554, креды NVR), минуя ретранслятор NVR.
Иначе — откат на NVR (с потерями).

**Ключевое уточнение (по живому аудиту 2026-06-16):** камеры на этом объекте —
**смешанные**. Часть напрямую доступна, часть нет. Поэтому прямой захват нельзя
включать «всем подряд» — он должен быть **с проверкой доступности**. Именно это
уже реализовал второй разработчик: `apply_camera_ips` теперь зондирует каждый IP
на :554 (`camera_import.filter_reachable`) и ставит `Camera.ip` только если камера
ответила; иначе — `None` (откат на релей). Это правильный, основанный на данных
подход — его и надо использовать.

**Живой аудит доступности камер (с рабочей сети, по IP из `dss.db.bak`):**

| диапазон | результат | вывод |
|---|---|---|
| `192.168.20.x` (камеры nvr15) | **17 из 19 отвечают** на 554+80, у каждой свой MAC Dahua | прямой захват возможен → путь с 0 потерь |
| `192.168.23.x` (28 шт., nvr58) | **0 из 28** | устаревшие IP старого объекта — мёртвые, чистить на релей |

**Что это значит для действий:**

1. Запустить **импорт IP с проверкой доступности** по каждому NVR:
   `POST /api/v1/nvrs/{nvr_id}/import-camera-ips` (кнопка «Обновить IP камер»).
   Он сам поставит 17 доступных камер nvr15 на прямой захват, а 28 мёртвых
   `.23.x` очистит на релей.
2. **Снова включить nvr15** (сейчас `enabled=0`) — после импорта его main-каналы
   получат путь с 0 потерь.
3. **Не опираться на `.netcheck/clean-nvr15.py`**: он жёстко очищает IP `.20.x`
   у nvr15, считая их недоступными — но они доступны (проверено вживую). Скрипт
   загоняет рабочие камеры обратно на лоссовый релей. Импорт с проверкой
   доступности делает его ненужным.

**Проверка №1 — заполнены ли IP в БД** (на проде, где лежит `dss.db` или Postgres):

```sql
SELECT n.id, count(c.id) cams, count(c.ip) with_ip
FROM nvrs n LEFT JOIN cameras c ON c.nvr_id=n.id
GROUP BY n.id;
```

**Проверка №2 — достаёт ли сервер камеру напрямую** (с сервера MediaMTX):

```bash
# реальный доступный IP из аудита выше
ffprobe -rtsp_transport tcp "rtsp://admin:***@192.168.20.101:554/cam/realmonitor?channel=1&subtype=0"
```
Если таймаут — камера за PoE/в изолированном сегменте → останется на релее
(импорт сам это определит). Если поток чистый — прямой захват даст 0 потерь.

### 1.2. Настройки на NVR/камерах Dahua (вне кода DSS)

Даже там, где main вынужденно идёт через NVR, сделать на самих устройствах:

- **Отключить SmartCodec (H.264+/H.265+)** на main — он ломает стандартное
  воспроизведение и усиливает фризы.
- Зафиксировать **CBR-битрейт** main в разумных пределах.
- **Укоротить интервал I-кадра (GOP)** примерно до 1×FPS — короче фриз при потере кадра.
- Обновить **прошивку** NVR/камер.
- Поднять лимит «исходящей/удалённой полосы» в настройках NVR, если он есть.

### 1.3. Сетка = sub, полный экран = main

Уже реализовано во фронтенде (`web/js/streams.js`, выбор tier). Оставить как есть —
это правильно: 64 main одновременно ни один NVR не вытянет.

### 1.4. Остаточная брешь: IP камер не обновляются автоматически

Второй разработчик уже снял **острую** часть: импорт теперь с проверкой
доступности и **чистит мёртвые IP** на релей (стойкий «мёртвый адрес» больше не
держится вечно). Но `apply_camera_ips` по-прежнему запускается только **вручную
или при создании NVR**. Если камера сменила IP по DHCP — до следующего ручного
импорта её main будет на релее (или на старом адресе). **Рекомендация:**
периодический фоновый рефреш IP (раз в N часов) или рефреш по событию
«main-путь не поднялся».

### 1.5. Инструмент тюнинга кодеков: `python -m app.fleet_tune`

Добавлен в этой сессии. Аудит/нормализация **main-кодека** камер: отключает
SmartCodec (через `BitRateControl=CBR`) и ставит `GOP=FPS`. Работает **только с
напрямую доступными камерами** (по аудиту — те самые 17 камер nvr15 на `.20.x`,
у них открыт порт 80). Камеры за PoE он не достанет — их кодек правится через
веб NVR. По умолчанию — только аудит; запись по `--apply`; тест на одной камере:
`python -m app.fleet_tune --only 192.168.20.101 --apply`.

## Уровень 2 — Настройка MediaMTX / хоста

Топология DSS: **вход RTSP по TCP**, **выход WebRTC (WHEP) / HLS** многим зрителям.
Важно подбирать ровно те ручки, что относятся к этой топологии.

- `writeQueueSize` (`mediamtx.yml:27`, сейчас `4096`) — буфер на **сторону
  зрителей**. Симптом нехватки — строка `reader is too slow` в логах MediaMTX.
  **Действие:** сначала проверить логи; поднимать (8192/16384) только если строка есть.
  Слепо ставить 65536 не нужно — это RAM на каждого зрителя.
  ```bash
  # есть ли вообще проблема на стороне fan-out:
  grep -i "reader is too slow" <лог mediamtx>
  ```
- `udpReadBufferSize` / `net.core.rmem_max` — **не относится** к вашему случаю:
  это буфер приёма **UDP**, а вход RTSP у вас TCP. Понадобится только если
  переключите RTSP-вход на UDP (не нужно).
- `rtspTransports: [tcp]` (`mediamtx.yml:37`) — **оставить TCP**. Это правильно:
  не маскирует потери под «успех», а замеры уже доказали, что потери на источнике.
- Таймауты on-demand (`backend/app/settings.py`): `main_start_timeout=20s`,
  `main_close_after=60s`, `sub=10s/30s`. Если NVR медленно поднимает источник —
  поднять `*_start_timeout`. Если зрители часто заходят/выходят и NVR не любит
  частые открытия — **уменьшить** `*_close_after`. Трогать только при конкретном
  симптоме в логах.

> Вывод по Уровню 2: **менять значения вслепую не нужно.** Это «наблюдать и
> подстраивать по логам», а не «поставить большие числа».

## Уровень 3 — Ревью кода: устойчивость (найденные дефекты)

Приоритезировано по влиянию. Файлы: `web/js/streams.js`, `web/js/grid.js`,
`web/js/config.js`, `backend/app/services/source_watch.py`,
`backend/app/services/path_sync.py`, `backend/app/services/mediamtx_api.py`,
`backend/app/db.py`.

### CRITICAL

- **C1. Массовое переподключение обходит лимит параллелизма (шторм).**
  `reconnectAllVisible()` и diff-рендер вызывают `connectCamera()` для всех тайлов
  (до 64). Лимит `maxConcurrent=8` ограничивает только согласование, но не темп
  открытий on-demand на NVR. Несколько операторов после сетевого моргания →
  синхронный залп открытий к одному NVR.
  **Фикс:** разносить массовые переподключения во времени (джиттер на тайл),
  ограничивать темп постановки в очередь, а не только параллелизм слива.

- **C2. Петля: фронтенд-ретраи ↔ watchdog авто-отключает NVR ↔ ретраи мёртвого пути.**
  Watchdog (`source_watch.py:178`) считает путь «активным» при наличии `source`
  (попытки источника). Сами ретраи фронтенда держат путь «активен-но-не-готов» →
  watchdog через ~6 с отключает NVR и **удаляет его пути**. Фронтенд об этом не
  знает и продолжает долбить удалённый путь (404) вечно. Усугубляется тем, что
  `conn.failures` сбрасывается в 0 на каждом ре-рендере, поэтому потолок
  `maxRetries=3` не работает.
  **Фикс:** (а) трактовать WHEP **404** как «путь удалён бэкендом» — прекратить
  ретраи медиа и опрашивать список камер на ре-активацию; (б) не сбрасывать
  `conn.failures`; (в) watchdog не должен принимать ретраи клиента за признак
  отказа аккаунта (ориентироваться на 401 в логах MediaMTX).
  **✅ Частично исправлено** (коммит `e71b96a`, второй разработчик): watchdog
  теперь не отключает NVR, который «только что стримил нормально», при
  одновременном падении всех каналов (`nvr_last_ready`) — трактует как
  транзиентный всплеск потерь, а не лок аккаунта. Это снимает самую опасную часть
  петли (NVR больше не «занулялся» под потерями). Остаётся часть фронтенда:
  (а) обработка 404 и (б) сброс `conn.failures`.

### HIGH

- **H1. Устаревшие WHEP-запросы держат слот очереди до 10 с.** При быстрых
  ре-рендерах/листании старые `doConnect` занимают слот до таймаута, новые тайлы
  висят в «connecting». **Фикс:** хранить `AbortController` на соединении и
  прерывать запрос при вытеснении/отключении.

- **H2. Утечка скрытых `<video>` при предзагрузке страниц.** Предзагруженные
  элементы в `document.body` не всегда удаляются на переходах → за часы патруля
  растут память и хэндлы декодера → тормоза. **Фикс:** явный реестр предзагрузок
  и подметание на каждом рендере.

- **H3. Reconcile 168 камер = 336 последовательных HTTP с таймаутом 5 с, без
    общего дедлайна, и вызывается прямо внутри 3-секундного цикла watchdog.**
  На холодном/нагруженном MediaMTX один reconcile может тянуться десятки секунд,
  блокируя watchdog. **Фикс:** (а) распараллелить с `asyncio.Semaphore(10–20)`;
  (б) из watchdog удалять только два пути камеры, не гонять полный reconcile;
  (в) общий бюджет времени на reconcile.

- **H4. Нет STUN/TURN — операторы из другой подсети не подключаются и уходят в
    вечный шторм ретраев.** `iceServers: []` (`config.js:14`) даёт только host-
  кандидаты. Для маршрутизируемых клиентов ICE падает, рабочий HLS-фолбэк при
  этом **не пробуется** (см. M5). **Фикс:** сделать `iceServers` настраиваемым
  (STUN, при необходимости TURN); при провале ICE на хост-кандидатах — быстро
  падать в HLS, а не зацикливать переподключения.

### MEDIUM

- **M1. Детектор зависаний пропускает «замороженный, но буферизованный» кадр.**
  Условие требует `readyState < 3`, поэтому фриз при `readyState=4` (зелёный
  индикатор на мёртвом потоке) **не ловится**. **Фикс:** считать зависанием
  остановку `currentTime` на `STALL_THRESHOLD` независимо от `readyState`.

- **M2. Ложные срабатывания детектора в фоновой вкладке** (троттлинг таймеров) →
  залп переподключений при возврате фокуса. **Фикс:** пропускать проверки при
  `document.hidden`, сбрасывать базу на `visibilitychange`.

- **M3. Пул БД не настроен под число операторов.** Дефолт async-пула (5+10) +
  отдельные сессии watchdog → исчерпание под нагрузкой, запросы виснут на
  `pool_timeout`. **Фикс:** явные `pool_size`/`max_overflow` для Postgres; reconcile
  убрать с критического пути watchdog (см. H3).

- **M5. WebRTC считается успешным до фактического ICE.** `tryWebRTC` возвращает
  `true` сразу после `setRemoteDescription`, поэтому HLS-фолбэк не срабатывает
  именно тогда, когда он нужен (off-segment операторы). **Фикс:** считать WebRTC
  успешным только при `iceConnectionState in {connected, completed}`.

### LOW

- **L1. Джиттер бэкоффа односторонний (±15% только вверх)** → стадо
  переподключается в узком окне. **Фикс:** «decorrelated jitter».
- **L2.** `rsplit("_ch", 1)` в разборе имён путей хрупок к id с `_ch`.
- **L3.** `list_paths` с таймаутом 5 с может оборвать весь reconcile на медленном
  MediaMTX.

## Порядок внедрения

1. **Уровень 1** — запустить импорт IP **с проверкой доступности** по всем NVR
   (он сам решит direct/relay), **снова включить nvr15**. Затем на доступных
   камерах (17 шт. nvr15) выключить SmartCodec/поставить GOP — вручную через веб
   или `python -m app.fleet_tune --apply`. **Это даёт основной эффект.**
2. **Уровень 3 — фронтенд-часть C2 + H1** — обработка 404, не сбрасывать
   `conn.failures`, убрать шторм переподключений. (Watchdog-часть C2 уже сделана.)
3. **Уровень 3 — H3 + M3** — убрать блокировки reconcile/пула под нагрузкой.
4. **Уровень 2** — по логам подстроить `writeQueueSize` и таймауты.
5. **Уровень 1.4 / H4 / остальное** — авто-рефреш IP, STUN/TURN при необходимости.

## Как проверять, что стало лучше

- `ffprobe`/`netcheck.py` на доступной камере (`192.168.20.101`) — ожидаем
  `RTP потеряно → 0` на main (против потерь через релей).
- `curl http://localhost:9997/v3/paths/list` — у активной прямой камеры `source`
  указывает на IP камеры (`192.168.20.x`), а не на NVR.
- В логах MediaMTX нет `reader is too slow`; в логах бэкенда нет частых
  `source-watch: NVR auto-disabled`.
