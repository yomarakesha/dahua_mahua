# План аудита DSS: потери RTP на main-потоках

**Дата:** 2026-06-10
**Статус:** черновик для обсуждения с коллегами
**Контекст:** DSS централизует main-потоки Dahua — 1 RTSP-вытяжка к NVR, fan-out к 3+ зрителям через MediaMTX. Проблема: на main-потоках массово теряются RTP-пакеты, камеры тормозят и тяжело открываются. Sub-потоки работают чисто.

---

## 0. Что уже известно (отправная точка)

Прогон `netcheck.py` (VPN выключен, изолированный MediaMTX, транспорт TCP, sourceOnDemand off):

| фаза | потоков | агрегат Mbps | Ethernet RX Mbps | RTP потеряно |
|---|---|---|---|---|
| 1 main | 1 | 1.0 | 1.3 | 4 977 |
| 2 main | 2 | 1.8 | 2.0 | 14 779 |
| 4 main | 4 | 2.8 | 3.2 | 32 423 |
| 8 main | 8 | 5.1 | 5.5 | 56 400 |
| 8 sub (база) | 8 | 3.9 | 4.3 | 156 |

**Предварительный вывод:**
- Линк (100 Мбит) загружен на ~5% — **это не ширина канала**. Гигабит сам по себе не поможет.
- Транспорт TCP → пакеты в пути не теряются. Дыры в RTP sequence = **NVR не отправил их, выкинул у себя на источнике**.
- Sub-потоков больше и битрейт выше, а потерь почти ноль → проблема именно в **отдаче main-потоков с NVR**.
- Даже один main теряет ~5000 пакетов, и битрейт всего ~1 Mbps (для Dahua main это аномально мало — обычно 4–8 Mbps) → NVR не вытягивает даже один полноценный main.

**Рабочая гипотеза:** узкое место — RTSP-сервер NVR (вся воронка main-потоков идёт через одно устройство, оно захлёбывается на крупных I-frame).

**Главный нерешённый вопрос аудита:** виноват именно NVR-как-ретранслятор, или сама связка камера→кодек→сеть? Это решает следующий тест (см. §3.1).

## 2. Что нужно посчитать (метрики и формулы)

Для каждого сценария считаем по данным MediaMTX API (`bytesReceived` дельта) + лог (`RTP packets lost`) + счётчики NIC:

- **Эффективный битрейт потока** = `Δbytes × 8 / Δt`. Сравнить с заявленным битрейтом камеры. Если эффективный << заявленного → поток режется (потери).
- **Доля потерь** ≈ `lost_pkts / (lost_pkts + received_pkts)`. Нужно вытащить received-пакеты (можно из RTCP/статистики MediaMTX либо оценить по битрейту и среднему размеру пакета ~1200 байт).
- **Скорость роста потерь на поток** — растут ли потери линейно с числом потоков (агрегатный лимит NVR) или есть фикс-потеря на каждый main независимо (пер-канальная проблема). Из таблицы §0 видно: потери есть даже на 1 потоке И растут с числом → похоже на оба фактора.
- **Точка перегиба по линку:** при какой агрегатной нагрузке Ethernet RX упирается в ~85–95 Мбит. Сейчас далеко (5.5 из 100) → линк не виноват.
- **Бюджет пропускной способности:** `Σ(битрейт main × число одновременных main)` vs заявленный исходящий лимит NVR vs скорость аплинка NVR. Это даёт теоретический потолок.
- **CPU/декодер NVR** во время пиковой отдачи (если снимается) — упирается ли в 100%.

## 3. Какие замеры провести (эксперименты)

### 3.1 РЕШАЮЩИЙ: main напрямую с камеры vs через NVR
Самый важный тест. Тянем один и тот же main-поток двумя путями и сравниваем потери:
- **A:** через NVR — `rtsp://nvr/cam/realmonitor?channel=N&subtype=0` (как сейчас).
- **B:** напрямую с камеры — `rtsp://camera_ip/cam/realmonitor?channel=1&subtype=0`.

**Трактовка:**
- B чистый, A теряет → виноват NVR-ретранслятор → переходим на прямое подключение к камерам (см. §5).
- Оба теряют → проблема в камере/кодеке/сети → крутим настройки камеры (битрейт, GOP, кодек).
- *Требует:* IP хотя бы одной камеры на достижимом LAN. Если камеры за внутренним PoE NVR — тест невозможен, и прямое подключение тоже отпадает (тогда только настройки NVR/камер через NVR).

### 3.2 Масштабирование (уже сделано, повторить после правок)
1 → 2 → 4 → 8 main, фиксируем потери и Ethernet RX. Уже есть в §0. Повторять как regression после каждого изменения.

### 3.3 Влияние настроек камеры (если §3.1 показал «оба теряют»)
По одному меняем на тестовом канале и мерим:
- битрейт main вниз (напр. 4096 → 2048 Кбит),
- H.265 ↔ H.264, вкл/выкл Smart Codec,
- GOP / I-frame interval,
- FPS вниз.
Цель — найти, какой параметр убирает потери.

### 3.4 Транспорт
- TCP vs UDP вытяжка к NVR (UDP может вести себя иначе на бурстах I-frame).
- Проверить writeQueueSize / readBufferCount в MediaMTX.

### 3.5 Контроль сети
- Параллельно с потоками — счётчики ошибок на портах свитча и `Get-NetAdapterStatistics` (дропы на NIC ПК).
- Дуплекс и скорость линка (`Get-NetAdapter`): NVR умеет гигабит, а линк сидит на 100 Мбит — проверить, нет ли полудуплекса/рассогласования (даёт потери при низкой загрузке).
- iperf3 ПК↔(точка рядом с NVR), чтобы подтвердить, что сам линк держит заявленную скорость без потерь.

### 3.6 Контеншн — другие смотрящие (гипотеза §1.4)
Прогнать один и тот же набор потоков **дважды**: в рабочий пик (когда операторы/DSS/мобилки активно смотрят) и в «тихое» окно (ночь / всех отключить). Сравнить RTP-потери.
- Потери в пик есть, в тишине пропадают → **общий бюджет отдачи NVR исчерпан другими клиентами.**
- Потери одинаковы → контеншн ни при чём, причина в нашей нагрузке/пределе/линке.
- Доп. сигнал: число онлайн-пользователей и индикатор remote bandwidth в веб-морде NVR во время обоих прогонов.

## 4. Как это работает сейчас (архитектура DSS)

```
Камеры → NVR → [RTSP pull, 1 на поток] → MediaMTX → [WHEP/WebRTC fan-out] → N зрителей
```

- Backend (FastAPI) управляет путями MediaMTX через его API (:9997), хранит NVR/камеры в SQLite, пароли шифрует Fernet.
- MediaMTX тянет с NVR по RTSP (sourceOnDemand, transport TCP) и раздаёт всем зрителям один раз вытянутый поток (fan-out) — поэтому на NVR идёт 1 подключение на камеру независимо от числа зрителей. Это уже правильно.
- Фронт (vanilla ES-модули, без сборки): main на фуллскрине, sub в гриде; есть grace-таймер на ICE `disconnected` (4с), чтобы временные разрывы не вызывали каскад реконнектов.

**Где ломается:** вытяжка main с NVR. NVR не отдаёт main-поток целиком — теряет пакеты на источнике. Fan-out и фронт здесь ни при чём; они лишь раздают уже битый поток.

## 5. Как это будет работать после правок (варианты решения)

Расставлены по силе. Выбор зависит от результата теста §3.1 и от того, где живут камеры (§1.2).

### Вариант A — прямое подключение к камерам (главный, если камеры на LAN)
```
Камеры → [RTSP pull напрямую, 1 на камеру] → MediaMTX → fan-out → зрители
NVR остаётся под запись/архив.
```
- Нагрузка размазывается по N камерам вместо одного NVR — каждая отдаёт свой один поток, это её штатный режим.
- **Меняет цель проекта** (сейчас «1 RTSP к NVR»). Обсудить с коллегами: согласны ли уйти от NVR-как-источника для live.
- Требует: достижимость камер с сервера, креды камер, поддержка в DSS источника «камера» (в БД уже есть IP/каналы — оценить объём доработки).

### Вариант B — настройки NVR/камер (быстрый, если прямое нельзя)
- Поднять «remote bandwidth limit» на NVR.
- Включить Smart Codec (H.264+/H.265+), уменьшить GOP, при необходимости снизить битрейт main.
- Минус: либо костыль, либо потеря качества; потолок NVR никуда не девается.

### Вариант C — ограничить одновременные main (паллиатив)
- Меньше одновременных main, `main_close_after` 60с → 10с (быстрее освобождать), main только на фуллскрине.
- Снижает пик, но не лечит причину.

### Сеть (отдельно)
- Гигабит на аплинке NVR/сервера — поможет ТОЛЬКО если упрёмся в линк (сейчас нет). Держать в уме на будущее при росте числа потоков.

### Прод-развёртывание (отдельный разговор)
- Билд/деплой на сервер, статический IP в LAN камер/NVR, без VPN-перехвата маршрута (sing-box/TUN ломал доступ к `192.168.20.0/24`). На сервере VPN не нужен — маршрут к камерам должен быть прямым.

Цель порядка — **сначала проверить, починим ли NVR-путь (дёшево, не трогаем архитектуру), и уходить на прямое подключение к камерам, только если доказан потолок NVR.**
